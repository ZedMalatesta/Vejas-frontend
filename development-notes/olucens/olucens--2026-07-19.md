# Sprint 5: Branding, Guest Access & Mobile Fixes — 2026-07-19

- **What was done:**
1) Injectable product brand — new `APP_BRAND` InjectionToken (`core/brand.ts`) initialized once in `app.ts` root providers with `{ name: 'Vejas', logoUrl: 'icons/logo.svg' }`. Header (logo + name), footer copyright, room loader, room-card fallback cover and the document `<title>` all read the token instead of hardcoding the name — renaming the product is now a one-line change;
2) New wind logo (`public/icons/logo.svg`, "vėjas" = wind) shown in the header and used as the favicon (`index.html` link fixed — it still pointed to the removed `favicon.ico`);
3) Guest ("stranger") access, frontend part: `roomGuard` silently signs visitors in via `authService.signInAsGuest()` (`POST /auth/guest` → throwaway JWT with `roles: ['guest']`), header shows the guest name chip instead of Login/Sign up, guests cannot own rooms or edit playlists;
4) Token auto-refresh: `auth.interceptor.ts` retries a 401 once after `refreshSession()` (single-flight), socket auth uses a token callback so reconnects pick up the fresh token;
5) Theme toggle in the header, persisted to `localStorage['vejas.theme']`;
6) Video player fixes: the YouTube IFrame API creates the `<iframe>` outside Angular's style encapsulation, so the scoped `.player-host iframe` rule never matched and the iframe stayed at its intrinsic 640×360 — fixed with `width/height: '100%'` in the `YT.Player` config plus a `::ng-deep` fallback. Replaced the magic `calc((100vh - 13rem) * 16/9)` wrapper width with a measured `--player-max-w` CSS variable (ResizeObserver on the component host + `document.body`, plus window resize) so the 16:9 box always fits the real space under the page chrome;
7) Mobile overflow fixes across all screens: `minmax(0, 1fr)` grid tracks (room layout, room-list grid), `min-width: 0` on flex/grid children (player column, chat, text-input host, profile cards), `overflow-wrap: anywhere` for long unbroken strings (room names, card name/description, chat authors), ellipsis on the room-card admin badge. Verified with a Playwright audit at 360×740, 390×844 and 1280×800 — zero horizontal overflow on all nine screens;
8) Housekeeping after porting: storage keys renamed to `vejas.*`, specs that mount branded components got an `APP_BRAND` test provider.

## **Problems:**
1) The room screen overlay (click shield, unmute button) never matched the actual video size — root cause was invisible in the styles because the bug lives in Angular's emulated encapsulation: elements created by third-party scripts get no `_ngcontent` attribute, so component-scoped selectors silently do nothing;
2) Mobile overflow had several independent causes stacking up (grid min-content sizing, flex min-width, unbreakable strings), so fixing one spot moved the overflow to the next one;
3) Guest access is a full-stack feature: the frontend half (guard, auth service, header) lives in this repo, but `POST /auth/guest` must exist on the backend. The legacy stub in `server/` and the current vejas backend do not implement it — until the endpoint is ported, visitors opening a room link will fail to auto-join.

- **Solutions:** Reproduced the layout bugs with a scripted Playwright audit (mocked API + fake JWT, measures `scrollWidth` vs viewport and lists offending elements per page) instead of resizing the browser by hand — this caught every overflow and proved the fixes. For the iframe, sized it via the player API config so the fix works regardless of encapsulation mode.

- **What I learned:** How Angular's emulated view encapsulation actually rewrites selectors and why third-party DOM needs `::ng-deep` or inline sizing; that CSS grid `1fr` means `minmax(auto, 1fr)` and follows min-content unless clamped with `minmax(0, 1fr)`; how to provide app-wide configuration through an InjectionToken initialized at the root component.

- **Plans:** 1) Port `POST /auth/guest` (and the guest checks in the rooms gateway) to the backend so guest access works end to end; 2) Remove the legacy stub `server/` once the real backend is deployed; 3) Consider deriving the storage-key prefix from `APP_BRAND` with a migration for existing keys;
